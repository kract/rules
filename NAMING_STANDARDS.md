## 基本ルール

- ファイル・フォルダ名（以下、「コンテンツ名」）は**半角英数字**と、本ドキュメントで指定する記号`()-_`のみを使用する。
- 本ドキュメントで指定する例外を除いて、2 バイト文字は使用しない。
- アプリによって自動生成されるコンテンツ名、及びキャメルケース（例：`camel case -> camelCase or CamelCase`）使用時以外は、全て小文字で命名する。ただし、シーン名などで`A`,`B`,`C`を使用する場合は必ず大文字とする。
- クライアントから受け取ったコンテンツ名については、変更せずそのまま使用する。2 バイト文字によって支障がある場合は、**複製して**英数字で命名し直す。
- 各階層において、使用しなくなったコンテンツがある場合は、その同階層に`(old)`フォルダを作って移動する。
- ISO 形式に従い、日付は`yyyy-mm-dd`、時刻は`yy-mm-ddTHHMMSS`（`:`はコンテンツ名として使用できないため省略する）で命名する。時刻は以下のコマンドを実行して生成する。

```bash
# 日時 （ex: 2025-03-22T06:18:53）
kract now
# 日付のみ (ex: 2025-03-22)
kract now --d
```

- 命名は https://www.abbreviations.com などを参考に、**分かり易さを損なわない範囲で**極力短くする。一般的でない省略法を使う場合は、単語の最初を抜き出すほうが好ましい。（頻出用語はドキュメント最後に記載）

```md
# Do

development -> dev
schedule -> sched

# Don't

shading -> sdg
```

- フォルダ構造は可能な限り浅く構築し、フォルダで細かく分割するよりは検索機能を使用することを優先する。

## プロジェクト ID

- 各プロジェクトには`アルファベット1文字+受注年（西暦下2桁）+通し番号（3桁）`を付与する。
- この ID は、後述のプロジェクトディレクトリ名・請求書 ID などとリンクしている。
- 既存のプロジェクトの続きとなるプロジェクトが発生した場合に新規の ID を付するかどうかは、既存のプロジェクトの内容をどの程度継承するかを鑑みて判断する。※軽微な修正の場合は以前の ID をそのまま使用しても良い。

## プロジェクトディレクトリ

- プロジェクトのルートディレクトリは、`プロジェクトID_直接やり取りするクライアント名_任意の名前`のフォーマットとする。
- プロジェクトのルートディレクトリの構造は以下の通りとする。
  - `comp`(composition) = プロジェクトファイル本体などの作業ファイルを格納する。
  - `in` = クライアントや外注先から受け取るファイルを格納する。
  - `out` = クライアントや外注先に共有するファイルを格納する。
  - `portfolio` = Web サイトなどポートフォリオ掲載のために使用するファイル群を格納する。（プロジェクト終了後に作成）

### `comp`ディレクトリの構造

- `comp`ディレクトリ内は、基本的にアプリごとにフォルダを作成する。
- Adobe アプリの場合は、そのアプリアイコンで使用されている 2 文字を小文字にして付与する。
  - `ae` = After Effects
  - `ai` = Illustrator
  - `ps` = Photoshop
- Adobe 以外のアプリで、比較的よく知られた略称が損座する場合はそれを付与する。
  - `b3d` = Blender
  - `c4d` = Cinema 4D
  - `md` = Marvelous Designer
  - `sbp` = Substance Painter
- 上記に当てはまらない場合は、適宜付与する。
  - `fusion` = Blackmagic Design Fusion
  - `resolve` = DaVinci Resolve
  - `unity` = Unity
  - `zbrush` = ZBrush

## 各アプリディレクトリの構造

### プロジェクト・シーンファイルのバージョニング

- 特別な理由がない場合は、コンテンツ名に 1 桁のバージョンを付する。バージョンが 2 桁担った場合はそのまま付する。
- 作業者がファイルを分岐したことを明示したい場合は、バージョンの後に分り易く名前をつける。これは後のバージョンにマージする。

### 画像・動画ファイルなどのバージョニング

- 特別な理由がない場合は、プロジェクト・シーンファイル名のバージョニング規則に従って、バージョンを合わせて命名する。

```text
design_v4.ai から PNG を書き出した場合は、design_v4.pngとなる。 design_v5.pngを作成したい場合は、それに合わせてファイルをバージョニングする。
```

- ae, c4d などから高頻度で書き出す場合は、時刻に基づく命名規則で行う（後述）

###`ae`ディレクトリの構造

- 各プロジェクトのワークフローに合わせて構築するが、以下を基本とする。
  - `export` = AE からレンダリングしたファイルを格納する。
  - `footages` = プロジェクトファイル外の、AE で使用するフッテージを格納する。
  - `lut` = Apply Color LUT エフェクト内で使用する LUT を格納する。
  - 今後使用しないコンテンツについては、基本ルールに則り`(old)`フォルダを作成、移動する。

### `ae/export`ディレクトリの構造

- コンポジション名と一致したフォルダ名を作成し、その中に mp4, mov ファイルなどを格納する。
- mp4, mov のファイル名は `コンポジション名_yyyy-mm-ddTHHMMSS.拡張子`で統一する。
- ae では以下のトークンを使用し、AE プロジェクトのルートディレクトリを指定してレンダリングする。

```bash
comp/ae/export/[compName]/[compName]_[dateYear]-[dateMonth]-[dateDay]T[timeHour][timeMins][timeSecs].[fileextension]
```

### `b3d`ディレクトリの構造

- 基本的に`b3d`内に直接`.blend`ファイルを配置することはせず、各プロジェクトのワークフローに合わせて役割やシーン毎に短い命名でディレクトリを作成し、その中に`.blend`ファイルを配置する。
  - `character` = キャラクター
  - `fx` = エフェクト
  - `environment` = 背景
  - - 今後使用しないコンテンツについては、基本ルールに則り`(old)`フォルダを作成、移動する。

### `b3d`各ディレクトリ内

- 各プロジェクトのワークフローに合わせて構築するが、以下を基本とする。
  - `render` = レンダーデータ
  - `tex` = テクスチャ

### `render`ディレクトリ内

- b3d プロジェクトと同名のディレクトリを作成し、必要に応じてテイク名などで分類する。
- デフォルトのレンダートークンは以下の通りとする。

```bash
//render/$prj/$prj_$take_####
```

### その他アプリケーションの構造

- 基本的には、各アプリのワークフローに合わせて分りやすく構築する。
  - `export` = 書き出しファイル ※書き出すファイルが複数のファイルタイプに及ぶ場合は、各ディレクトリを作成して書き出し先としても良い。

## `in`,`out`ディレクトリの構造

- `yyyy-mm-dd_内容`を基本とする。
- 継続して頻繁にデータをやり取りするクライアント・外注先がある場合は、別途その名前のディレクトリを作成しても良い。その場合、頭に`_`を付する。
- 受け取り・受け渡し先を明示したい場合は、`yyyy-mm-dd_from名前_内容`、`yyyy-mm-dd_to名前_内容`のように命名する。
- `in`ディレクトリは、クライアントや外注先から受け取ったコンテンツを**そのままの状態**で保存する。
  - ZIP ファイルで受け取った場合は、解答後の ZIP ファイルもそのまま保持する。（上書き削除されてしまった場合に元の状態に簡単に戻せるように）
  - 2 バイト文字を含んでいても、名前は変えない。（クライアント・外注先との意思疎通のため）
